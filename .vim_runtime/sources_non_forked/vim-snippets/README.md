<h2>Snippet engines supporting vim-snippets</h2>

<p>There are different forks of snippet engines which allow the user to insert
snippets by typing the name of a snippet hitting the expansion mapping.</p>

<ul>
<li><a href="http://github.com/SirVer/ultisnips">github.com/SirVer/ultisnips</a>:
python, supports all snippets in this repo.</li>
<li><a href="https://github.com/garbas/vim-snipmate">github.com/garbas/vim-snipmate</a>:
VimL, snipmate-snippets, engine sometimes behaves strange. Supports
snippets/*</li>
<li><a href="https://github.com/Shougo/neosnippet.vim">github.com/Shougo/neosnippet</a>:
VimL, supports snippets/* with some configuration.</li>
<li><a href="https://github.com/drmingdrmer/xptemplate">github.com/drmingdrmer/xptemplate</a>:
Totally different syntax, does not read snippets contained in this file, but
it is also very powerful. It does not support vim-snippets (just listing it
here for completeness)</li>
</ul>

<p>There tries to be a more comprehensive list (which still is incomplete) here:
<a href="http://vim-wiki.mawercer.de/wiki/topic/text-snippets-skeletons-templates.html">http://vim-wiki.mawercer.de/wiki/topic/text-snippets-skeletons-templates.html</a></p>

<p>UltiSnips has additional features such as high speed, nesting snippets,
expanding snippets in snippets and offers powerful transformations on text in
snippets (like visual selections or placeholder texts).</p>

<p>Which one to use? If you have python give <a href="http://github.com/SirVer/ultisnips">SirVer/ultisnips</a> a try because
its fast and has the most features.</p>

<p>If you have VimL only (vim without python support) your best option is using
<a href="https://github.com/garbas/vim-snipmate">garbas/vim-snipmate</a> and cope with the
minor bugs found in the engine.</p>

<p><strong>Q</strong>: Should snipMate be deprecated in favour of UltiSnips?</p>

<p><strong>A</strong>: No, because snipMate is VimL, and UltiSnips requires Python.
Some people want to use snippets without having to install Vim with Python
support. Yes - this sucks.</p>

<p>One solution would be: Use snippets if they are good enough, but allow overriding them
in UltiSnips. This would avoid most duplication while still serving most users.
AFAIK there is a nested-placeholder branch for snipMate too. snipMate is still
improved by Adnan Zafar. So maybe time is not ready to make a final decision yet.</p>

<p><a href="https://github.com/honza/vim-snippets/issues/363">github issue/discussion</a></p>

<h2>Vendor Snippets</h2>

<p>Additional library and framework snippets are available for UltiSnips users in
the <code>UltiSnips/</code> directory. These files are removed from the default language
namespaces to prevent them from all being loaded automatically. If there is a
separate library, framework, or package you would like to support open a pull
request!</p>

<p>Additional snippets can be added to the current buffer with the
<code>:UltiSnipsAddFiletypes</code> command followed by the snippet name without the
&ldquo;snippets&rdquo; ending. For example, to add the JavaScript Jasmine snippets, run:
<code>:UltiSnipsAddFiletypes javascript-jasmine</code>. To have this snippet loaded
everytime a JavaScript file is opened or created you can add the command to your
 -<code>.vim/ftplugin/javascript.vim</code> file. Another way is to add
 <code>autocmd FileType js UltiSnipsAddFiletypes javascript-jasmine</code> in your <code>.vimrc</code>.</p>

<p>For more see the UltiSnips docs (<code>:help UltiSnips</code>).</p>

<h2>Things to consider when contributing</h2>

<p>Some snippets are useful for almost all languages, so let&rsquo;s try to have the same
triggers for them:</p>

<pre class="chroma"><code><span class="k">if</span> : <span class="k">if</span> without <span class="k">else</span>
ife: <span class="k">if</span> <span class="nv">$1</span> <span class="k">else</span> <span class="nv">$2</span>
eif : <span class="k">else</span> <span class="k">if</span> <span class="o">(</span><span class="nv">$1</span><span class="o">)</span> <span class="o">{</span> .. <span class="o">}</span>
el  : <span class="k">else</span> ..
wh  : <span class="k">while</span> <span class="o">(</span>cond<span class="o">)</span> ...</code></pre>

<p>Don&rsquo;t add useless placeholder default texts like:</p>

<pre class="chroma"><code><span class="k">if</span> <span class="o">(</span><span class="si">${</span><span class="nv">1</span><span class="p">:</span><span class="nv">condition</span><span class="si">}</span><span class="o">){</span>
  <span class="si">${</span><span class="nv">0</span><span class="p">:</span><span class="nv">some</span><span class="p"> code here</span><span class="si">}</span>
<span class="o">}</span></code></pre>

<p>instead use:</p>

<pre class="chroma"><code><span class="k">if</span> <span class="o">(</span><span class="si">${</span><span class="nv">1</span><span class="si">}</span><span class="o">){</span>
  <span class="si">${</span><span class="nv">0</span><span class="p">:</span><span class="si">${</span><span class="nv">VISUAL</span><span class="si">}}</span>
<span class="o">}</span></code></pre>

<p>Exception: Functions which are used less often, such as Vim&rsquo;s <code>matchall()</code>, <code>matchstr()</code>
functions which case hints may be helpful to remember order. In the VimL case
get vim-dev plugin which has function completion</p>

<p>Thus for conditions (while, if ..) and block bodies just use ${N} - Thanks</p>

<p>When the snippet can be used to wrap existing code leverage <code>${VISUAL}</code></p>

<p>Open questions:
What about one line if ee then .. else .. vs if \n .. then \n &hellip; \n else \n .. ?
Which additional policies to add?
Discuss at: <a href="https://github.com/honza/vim-snippets/issues/230">https://github.com/honza/vim-snippets/issues/230</a></p>

<p><em>folding markers</em>:
Until further work is done on <code>vim-snipmate</code>, please don&rsquo;t add folding markers
into snippets. <code>vim-snipmate</code> has some comments about how to patch all snippets
on the fly adding those.</p>

<p>Currently all snippets from UltiSnips have been put into UltiSnips - some work
on merging should be done (dropping duplicates etc). Also see engines section above.</p>

<li>add snippets/ruby-1.8.snippets (1.8 only)</li>
<li>add snippets/ruby-1.9.snippets (1.9 only)</li>
</ul>

<p>then configure <a href="https://github.com/garbas/vim-snipmate">https://github.com/garbas/vim-snipmate</a> this way:</p>

<pre class="chroma"><code><span class="k">let</span> <span class="nx">g</span>:<span class="nx">snipMate</span> <span class="p">=</span> {}<span class="err">

</span><span class="err"></span><span class="k">let</span> <span class="nx">g</span>:<span class="nx">snipMate</span>.<span class="nx">scope_aliases</span>[<span class="s1">&#39;ruby&#39;</span>] <span class="p">=</span> <span class="s1">&#39;ruby,ruby-rails,ruby-1.9&#39;</span></code></pre>

<p>If it happens that you work on a project requiring ruby-1.8 snippets instead,
consider using <code>vim-addon-local-vimrc</code> and override the filetypes.</p>

<p>Well - of course it may not make sense to create a new file for each
ruby-library-version triplet. Sometimes postfixing a name such as</p>

<pre class="chroma"><code>migrate_lib_20_down
migrate_lib_20_up</code></pre>

<p>will do it then if syntax has changed.</p>
